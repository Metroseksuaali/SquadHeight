# SquadHeight — project context for Claude

## What this is

Exports "true surface" heightmaps from the Squad SDK (UE5 editor, Python
scripting) for [SquadCalc](https://github.com/sh4rkman/SquadCalc): top-down
line-trace grid over each map's collision, so buildings/bridges/rocks are
included and foliage is excluded. README.md covers usage; this file covers
what a working session needs that the README doesn't.

## Layout in one glance

* `tools/export_heightmap.py` — the exporter (runs inside the UE editor;
  `CONFIG` dict at top). `tools/batch_export.py` + `run_batch_export.bat` —
  headless batch over all maps in `tools/maps_config.json` (generated by
  `tools/make_config.py`; bounds come from `tools/squadcalc_bounds.json`).
* `output/<Map>/` (gitignored) — `heightmap.json` (full-res 1 m),
  `heightmap_500.json` (SquadCalc drop-in), `heightmap.png` (16-bit gray),
  `meta.json` (bounds/resolution/orientation/z-offset + trace stats).
* `squadcalc-test/` — local SquadCalc in Docker: serves OUR heightmaps, proxies
  everything else to production `squadcalc.app/api`. `stage_heightmaps.py`
  copies output → `heightmaps/`; `crosscheck.py` / `render_compare.py` validate
  alignment against production data.
* `tools/build_release_zips.py` — packages GitHub Release assets (that's how
  data is distributed; it is never committed).
* Editor-side scripts import `unreal`; offline tools are stdlib-only except
  numpy (`find_alignment.py`, squadcalc-test checks) and Pillow (doc renders).

## Hard rules

* **Never commit exported data** — it derives from Squad © Offworld
  Industries. `output/`, staged heightmaps and compare PNGs are gitignored;
  distribution happens via GitHub Releases (MIT covers code only, see LICENSE).
* **No machine-specific paths or personal info in committed files** — paths go
  in `settings.bat` (gitignored, template `settings.example.bat`). Everything
  committed is in English.
* **`meta.json` + `heightmap.json` are a published data contract** with
  downstream consumers (SquadCalc delivery; sibling project
  `O:\vibecode\SquadSight` documents it in its `docs/DATA_FORMAT.md`). Don't
  rename/remove fields casually.
* **Always take `bounds_m` from `tools/squadcalc_bounds.json`** (extracted from
  SquadCalc's `maps.js`) — auto-detected landscape bounds do NOT match the
  minimap square.

## SDK gotchas (each cost real debugging — don't rediscover them)

1. Commandlets don't load streaming sublevels; `batch_export.py` attaches them
   via the asset registry, with a blacklist of non-geometry variants
   (`_SUBLEVEL_SKIP`: gameplay/lighting/whitebox/reference/...).
2. Hidden editor levels have NO collision — masters like Mutaha/Narva load
   sublevels hidden; they're forced visible (`set_levels_visibility`).
3. Invisible Box/Sphere/Capsule/Brush shapes are excluded **only when the actor
   has no static meshes** — old-style buildings use box collision shells, and
   excluding all boxes erased Chora's houses.
4. `BP_POI_Reference*` designer markers float ~40 m above POIs — excluded by
   actor-class prefix.
5. The editor OOMs after ~8 large maps — the .bat relaunch-loops and resumes
   (maps with `meta.json` are skipped; `SQUADHEIGHT_FORCE` overrides).
6. Hole filling: bounded frontier walk for seam cracks, min-fill for large
   empty regions (a full Python set of 11M holes once OOM'd the editor).
7. Newer/reworked maps live in game-feature plugin roots (`/Al_Basrah/`,
   `/Harju/`, `/BlackCoast/`, `/SanxianIslands/`), not `/Game/Maps`;
   `/Game/Maps/BASRAH_CITY` is a pre-rework leftover — never export it.
8. This UE build has no `GameplayStatics.break_hit_result`; hit parsing
   auto-detects a strategy (this SDK: `to_tuple()`).

## Data semantics worth remembering

* Heights are meters, min-normalized to 0; `world_z = value + z_offset_m`.
* Water and out-of-play ground read 0 — correct for mortar math. Maps whose
  landscape doesn't fill the minimap square (notably Chora) have zero-filled
  out-of-play borders; the playable area is accurate. Don't "fix" the borders
  by inventing terrain.
* Jensen/Kamdesh/Pacific legitimately have few structures (gameplay-spawned
  props / instances that don't register); legacy data was terrain-only anyway.
* SquadCalc hardcodes heightmap width 500 (`squadHeightmaps.js`) and has no
  heightmap data in its repo — it fetches `${API_URL}/img/maps/<map>/heightmap.json`
  from its API server, which only its author controls. Shipping our data means
  delivering files to him (or testing locally via `squadcalc-test/`).

## Diagnostics when something looks wrong

`tools/diag_map.py` (level inventory + test rays, env `SQUADHEIGHT_DIAG_MAP`),
`tools/probe_points.py` (which actor a ray hits at given XY, env
`SQUADHEIGHT_PROBE`), `meta.json`'s `hit_assets_top` histogram (spot foliage
leaks / weird actors), `squadcalc-test/render_compare.py` (ours vs production
side by side).
